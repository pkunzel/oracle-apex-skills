# APEX_AUTHENTICATION

Source: [Oracle APEX 26.1 API Reference](https://docs.oracle.com/en/database/oracle/apex/26.1/aeapi/APEX_AUTHENTICATION.html)

Status: curated first-pass reference.

## Purpose

`APEX_AUTHENTICATION` exposes server-side helpers for authenticating the current APEX session, checking whether the current request is authenticated or public, integrating external identity providers, handling SAML callbacks, and managing persistent-authentication/user-name cookies.

Most applications should use declarative authentication schemes. Use this package when a custom scheme, authentication plug-in, login process, logout link, external identity-provider callback, or test harness needs the documented runtime hooks.

## When To Use

Use `APEX_AUTHENTICATION` when:

- A custom login page must call `LOGIN` after collecting user credentials.
- An authentication plug-in validates identity outside APEX and must call `POST_LOGIN`.
- Server-side code needs to branch on `IS_AUTHENTICATED` or `IS_PUBLIC_USER`.
- An external OAuth/OIDC-style flow needs a callback URL generated by `GET_CALLBACK_URL`.
- SAML service-provider metadata or callback endpoints are needed.
- A login page must remember the user name with explicit consent and instance settings allow it.
- Persistent authentication entries need to be removed for the current browser or a specific session/user.

Do not use this package to bypass declarative authorization. Authentication answers "who is this user"; use `APEX_AUTHORIZATION` or `APEX_ACL` to answer "what can this user do".

## Authentication Checks

Use these helpers inside a normal APEX request when a process or plug-in needs to behave differently for public and logged-in users.

```sql
begin
    if apex_authentication.is_authenticated then
        apex_debug.info('Authenticated user: %s', :APP_USER);
    else
        apex_debug.info('Public request');
    end if;
end;
/
```

`IS_PUBLIC_USER` returns the inverse perspective: true when the current session is not authenticated.

## Login

Signature:

```sql
apex_authentication.login(
    p_username            in varchar2,
    p_password            in varchar2,
    p_uppercase_username  in boolean default true,
    p_set_persistent_auth in boolean default false );
```

Simple custom login page process:

```sql
begin
    apex_authentication.login(
        p_username            => :P9999_USERNAME,
        p_password            => :P9999_PASSWORD,
        p_uppercase_username  => false,
        p_set_persistent_auth => :P9999_REMEMBER_ME = 'Y');
end;
/
```

Use `p_set_persistent_auth` only when the application explicitly supports "remember me" behavior and instance-level persistent authentication is enabled.

## Post-Login For External Validation

`POST_LOGIN` is for schemes where credentials or identity have already been validated outside APEX. It authenticates the current session without re-performing the standard credential check.

```sql
begin
    -- Assuming an upstream identity provider has already validated the request
    -- and mapped it to the local application user name in l_username.
    apex_authentication.post_login(
        p_username           => l_username,
        p_password           => null,
        p_uppercase_username => false);
end;
/
```

Never call `POST_LOGIN` simply because a request contains a user name. Only call it after a trusted validation step has succeeded.

## Logout

Signature:

```sql
apex_authentication.logout(
    p_session_id in number,
    p_app_id     in number );
```

Logout from the current browser session:

```sql
begin
    apex_authentication.logout(
        p_session_id => apex_application.g_instance,
        p_app_id     => apex_application.g_flow_id);
end;
/
```

The documented behavior closes the session and redirects to the application's home page. It is intended to be invoked from the browser request flow.

## External Callback URLs

`GET_CALLBACK_URL` returns a browser URL for external login pages. Optional `p_x01` through `p_x10` values are passed back to the authentication plug-in callback.

```sql
declare
    l_callback_url varchar2(4000);
begin
    l_callback_url := apex_authentication.get_callback_url(
        p_x01 => :APP_SESSION,
        p_x02 => :P9999_RETURN_TO);
end;
/
```

Assuming an authentication plug-in redirects the user to an external identity provider, store only non-secret correlation values in the callback parameters. Use server-side state for secrets and verifier data.

`CALLBACK` and `CALLBACK2` are landing resources for external login pages and should normally be called by the browser or identity provider, not by ordinary page processes.

## SAML

Use `SAML_METADATA` to emit service-provider metadata for an application or instance:

```sql
begin
    apex_authentication.saml_metadata(
        p_app_id => :APP_ID);
end;
/
```

`SAML_CALLBACK` is the landing resource invoked by the SAML identity provider. Do not hand-roll parsing of `SAMLResponse`, `RelayState`, or signatures in normal application code.

## Username Cookies And Persistent Auth

Read a remembered username:

```sql
declare
    l_username varchar2(255);
begin
    l_username := apex_authentication.get_login_username_cookie;
    :P9999_USERNAME := l_username;
end;
/
```

Send or clear the username cookie based on consent:

```sql
begin
    apex_authentication.send_login_username_cookie(
        p_username => :P9999_USERNAME,
        p_consent  => :P9999_REMEMBER_USERNAME = 'Y');
end;
/
```

Check instance-level capabilities before offering persistent-login behavior:

```sql
begin
    if apex_authentication.persistent_auth_enabled then
        apex_debug.info('Persistent authentication is available.');
    end if;
end;
/
```

Clear persistent authentication for the current browser:

```sql
begin
    apex_authentication.remove_current_persistent_auth;
end;
/
```

## Safety Guidance

- Prefer declarative authentication schemes unless implementing a custom authentication flow or plug-in.
- Do not treat client-supplied callback parameters as trusted identity.
- Use `APEX_AUTHORIZATION` or `APEX_ACL` for permissions after login.
- Avoid storing secrets in `p_x01` through `p_x10` callback parameters.
- Use explicit user consent for username cookies.
- Use generated member pages below for exact signatures, especially `CALLBACK`, `CALLBACK2`, and SAML procedures.
